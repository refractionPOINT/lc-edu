comms

Cost: $0.50 / sensor / month

Comms is the place to respond to detections as a team. It's built close to the metal to give full visibility of the actions your team is taking to mitigate threats.

When a detection happens, a Room is generated for your team to assign ownership, discuss, assess indicators of compromise, and issue commands. It's more than a paper trail - it's a digital war room.

If decisive action is required without time to explain what needs doing, take the initiative and your team will see the process in real-time. An additional benefit of this is newer members more easily learning proper protocol by witnessing what a real-time investigation and response looks like.

Rich linking of entities allows your team to observe trends across detections: like an endpoint with several detected threats, or a common detection popping up on several boxes. Navigate between them easily or merge rooms together to create a shared context for complex emerging issues.

Whether it be for auditing, shift changeover, or onboarding, a quick search through Comms will let you pick up where the last user left off with ease.

cuttingedge

Cost: $0.30 / sensor / month

The Cutting Edge threat feeds are aggregated, filtered and augmented from a multitude of sources. Emphasis is put on timely Indicators of Compromise (IoC) for new and upcoming threats.

Usage

When enabled, the cuttingedge service will automatically apply a managed D&R ruleset to your organization which will then generate detections.

Produced detections pre-empt investigation by including metadata about the origin of the IoC as well as the nature of the threat.

A sentiment metric is available to help dial in alerting thresholds.

dumper

Cost: Free

The Dumper service allows the dumping of several forensic artifacts on Windows hosts.

It supports a single action, which is to dump. It supports multiple targets, memory to dump the memory of the host and mft to dump the MFT of the file sytem to CSV. The Service then automates the ingestion of the resulting dump (and dump metadata) to LimaCharlie's Artifact Ingestion system where it can be downloaded or analyzed, and where you can create D&R rules to automate detections of characteristics of those dumps.

Usage

When enabled, dumper will be added to the Service Request view inside your organization. It will accept the following parameters:

  • sid - a sensor ID for the host to perform the memory dump
  • retention (optional) - the number of days to retain the stored results

On submission of a request, the dumper service performs a full memory dump of a host, uploads the resulting dumps to LimaCharlie's External Logs system, and deletes the local dumps afterwards. Dumper requests can also be made via D&R rules; check out the example service request for more details.

Notes

The dumper service does not currently validate that the host has enough available disc space for the memory dump.

Although the dumper service is free, the resulting memory dumps uploaded to LimaCharlie are subject to External Logs pricing.

exfil

Cost: Free

Automates management of event selection for events from agents that should be sent to the cloud in real-time.

By default, LimaCharlie sensors send events to the cloud based on a standard profile that includes events like NEW_PROCESS, DNS_REQUEST etc.

If you enable the Exfil Service, this default profile is replaced by a custom set of rules you define. The rule customization is done in two parts:

  1. The Event Rules, which simply define a list of events that should be sent to the cloud based on platform and tags.
  2. The Watch Rules, which define additional events that should be sent to the cloud based on the content of each individual event.

The Watch Rules allow you to specify a platform and tag to select which sensors the rule applies to, plus these elements:

  • Event: the specific event type that should be evaluated, like MODULE_LOAD.
  • Path: the path within the event component whose value should be evaluated, like FILE_PATH.
  • Operator: the type of evaluation/comparison that should be done between the value at Path in the event and the Value.
  • Value: the value used in the comparison.

Usage

When enabled, the Exfil Control view will be available within your organization. From there you can change the default set of events to be sent for each sensor platform or you can use tags to create more targeted configurations.

With exfil it's also possible to turn off collection of all events, minimizing usage.

For more information, check out the Exfil documentation.

pagerduty

Cost: Free

The PagerDuty service allows you to trigger events within PagerDuty. It requires setup of the PagerDuty access token in the Integrations view within your Organization.

Setup

PagerDuty's side must first be configured to receive the API notifications.

  1. In PagerDuty, go to the 'Integrations' tab of your Service
  2. Click 'Add a new integration'
  3. Give it a name, for example 'LimaCharlie'
  4. In the 'Integration Type' section, select the radio button 'Use our API directly' and select 'Events API v2' from the dropdown
  5. Click 'Add Integration'
  6. Back in the 'Integrations' page, you should see your new integration in the list. Copy the 'Integration Key' to your clipboard
  7. In LimaCharlie within the 'Integrations' view, set the PagerDuty API Key by pasting the key just generated
  8. Enable this service

reliable-tasking

Cost: Free

Enables reliable, asynchronous and large scale tasking of LimaCharlie sensors. This allows a task to be sent to a large number of sensors, even if they're offline. The service takes care of resending the task to those offline sensors, ensuring each of the targeted sensors is tasked at least once.

Usage

When enabled, reliable-tasking will be added to the Service Request view inside your organization. It will accept the following parameters:

  • task - a command line task (i.e. os_version)
  • sid - a sensor ID for the host to perform the memory dump
  • tag - a tag representing multiple sensors
  • ttl (optional) - the number of seconds to keep trying for; defaults to 1 week

On submission of a request, the reliable-tasking service will automatically send the task to sensors when they come online.

responder

Cost: Free

The Responder service is able to perform various incident response tasks, such as a sweep. A sweep is an in-depth look through the state of a host. The sweep will highlight parts of the activity that are suspicious. This provides an informed starting position when beginning an investigation. It allows focus to be on the important things right away.

The types of information returned by the sweep is constantly evolving but you can expect it to return the following information:

  • A full list of processes and modules
  • A list of unsigned binary code running in processes
  • Network connections with a list of processes listening and active on the network
  • Hidden modules
  • A list of recently modified files
  • Unique or rare indicators of compromise

sensor-cull

Cost: Free

The sensor-cull service performs continuous cleaning of old sensors that have not connected to your organization in a number of days.

This is useful for environments with cloud deployments or VM template based deployments that enroll sensors over and over.

Interacting

The sensor-cull service does not implement a web UI, but because of its simplicity all interactions, like managing rules can be done via the generic Service Request section of the web interface.

The service works by creating rules that describe when various sensors should be cleaned up.

Each rule specifies a single sensor tag used as a selector for the sensors the rule applies to. A rule also has a name which is helpful for bookkeeping, and a ttl which is the number of days a sensor can not connect to the cloud before it becomes eligible for cleanup.

Cleanup is automatically applied once a day.

Full documentation here.

sigma

Cost: Free

This service provides a core set of the open source Sigma rules in a managed fashion. It provides hundreds of rules and is a great boiler-plate rule pack to apply to your LimaCharlie deployment. Sigma is an open source format for describing signatures in a generic way so that they can be applied through multiple technologies (like LimaCharlie).

The Sigma project is available here: https://github.com/Neo23x0/sigma

The specific rules, converted and applied through this service are available here: https://github.com/refractionPOINT/sigma/tree/lc-rules/lc-rules

Some Sigma rules on Windows rely on Windows Event Logs which are not collected by LimaCharlie by default. In order to leverage these you will need to configure automated collection of the relevant Windows Event Logs through the Artifact Collection service.

YARA is a tool aimed at, though not limited to, helping malware researchers identify and classify malware samples. With YARA you can create descriptions of malware families, as well as anything else, based on textual or binary patterns. Each description, a.k.a rule, consists of a set of strings and a boolean expression which determine its logic.

zeek

Cost: Determined by usage

Zeek (formerly Bro) provides a comprehensive platform for network traffic analysis, with a particular focus on semantic security monitoring at scale.

Once enabled, the Zeek service will watch for PCAP files being ingested through the Artifact Ingestion system. For each PCAP, the Zeek service will run the Zeek tool on the PCAP. All the resulting Zeek log files will then be re-ingested into Artifact Ingestion where they can be viewed or Detection & Response rules can be created to generated detections or automate responses. The Zeek service pricing is based on the size of PCAPs processed: $0.02 per 1GB.

The Zeek documentation can be found here.
