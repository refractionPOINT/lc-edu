Dashboard

The Dashboard provides quick access to search all endpoint telemetry across the entire fleet for all time. It also displays a list of Services that are currently running.

The Dashboard is composed of three distinct sections.

Search

Using the Search field Users can search across telemetry for all endpoints in the fleet for all time across the following indicators of compromise (IOCs):
  • Sensor ID
  • Hostname Prefix
  • IP Address
  • Hash
  • File Path

Graphs

The Dashboard displays three different graphs which are updated every few hours and represent a rolling window over 30 days of time.

Sensors online over time displays a daily wrap-up of the number of sensors connected to the LimaCharlie Cloud. Hovering over this graph will display the date and number of events for the given point on the graph.

Events over time displays a daily wrap-up of the number of endpoint events sent to the LimaCharlie Cloud. Hovering over this graph will display the date and number of events for the given point on the graph.

Detections over time displays a daily wrap-up of the number of  Detection rules that were triggered. Hovering over this graph will display the date and a color-coded list of detections for the given point on the graph.

Jobs

The Jobs section displays a list of Services being executed in the LimaCharlie Cloud. Clicking on an individual line item will produce a modal showing the details of the job and and any output produced by it.

The labels for the Job line items is explained as follows:

Status indicates the current status of the job.

Time indicate the time of the last update.

Service displays the name of the Service that is currently running.

Hostname lists the name of the endpoint on which the Service is being performed. The majority of Services are enacted on a single endpoint but in certain circumstances this field can be empty when the Service applies to more than one endpoint.

Cause lists the reason the Service was initiated.

Data Visualization

Through the web application, LimaCharlie provides an interface where users can explore prevalence and timing for domains, IP addresses, files, file paths, hashes and user names across their entire fleet for up to a year’s worth of data. This console also allows users to search for a given sensor ID and deep dive all of the aforementioned data points for a single endpoint over the given time period.

When searched, an indicator of compromise will show up in the graph along with information relating to when it was observed, how often and across how many endpoints. Users can then click on any given graph node to get more information and start to drill down.

The Data Visualization page allows users to search for a variety of data types across the entire fleet for up to a year, as well as a detailed telemetry from any given endpoint for the given time range.

Data Types

The data types that can be searched for are:

  • Domain: Exact match for a given domain (www.google.com returns a different result than google.com)
  • SID: Search by sensor ID. Searching with this parameter will bring back details across all types for the given endpoint for the given time range.
  • IP: IP addresses that requests are made to and from.
  • File Path: Search for observed file paths.
  • File Hash: Search for observed file hashes.
  • User: Search by username.
  • File Name: Search by file name.
  • Service Name: Search by service name.

Node Size

The node - as it is displayed on the graph - is calculated based on how many events of the given type are bucketed into the given hour represented by the x axis (you can also hover over the node to get the time). These event are bucketed by type and represent events across multiple endpoints. Clicking on a node will bring up a list of the given events in the lower menu. Clicking on a listed event in the lower menu will bring up sensor details which can be further expanded into the tabbed region on the lower right.

Prevalence

The prevalence is represented by a given nodes position on the y axis. The prevalence represents the number of hosts the specific type instance that has been observed on during the time period.

Opacity

Opacity is related to the number of objects present in the given node in relation to the largest object returned for the given data set (this measure is meant to highlight outliers).

Information Panel

The information panel in the lower half of the screen is where information is exposed as you interact with graph.

Objects

Clicking on a node in the graph will populate this area with type and name(s) for the node.

Sources

Clicking on an object name will populate the Sources tab with the Source ID(s) and the first time that object was seen on the given source.

Tabbed Data

The tabbed data section shows details for a given source. The information is spread across three tabs as outlined below.

Source: when a Sensor ID is selected the Source tab will display the following data. It also provides a link into the Historical Insight UI at the time when the selected object was observed.

  • Hostname
  • Last Seen Timestamp
  • Enrollment Timestamp
  • External IP
  • Internal IP
  • Installer ID
  • Organization ID

Metadata: the metadata represents additional data specific to the selected object that may be useful. For example, the metadata for a domain would be the underlying IP address.

Data: the raw JSON telemetry for events at the time of the given event.






Sensors

The Sensors page provides a filterable list of all sensors in the current Organization’s fleet. This filter uses progressive text matching across hostnames.

From this list users can glean operational information from status icons and tags at a glance. Users may also interact with the endpoint directly or access the historical telemetry for the given endpoint across all time. 

Each line item representing a sensor contains the following information.

Hostname: the name of the host (or endpoint hostname).

Tags: a list of tags that are applied to the given host.

Status: consists of three different icons representing operational information about the endpoint.

The cloud icon indicates whether or not the sensor is online.


The lock icon indicates whether or not the endpoint has been put into a lockdown state.


The eye icon indicates whether or not there is kernel access available on the given endpoint.


Action: consists of three different icons providing various access to the endpoint, information about it and its historical telemetry.

The clipboard icon launches a modal with information about the endpoint and an interface for editing the tags associated with the given endpoint. This modal also has a button allowing users to isolate the given endpoint from the network while maintaining a line of command and control. Users may also delete the individual sensor using this modal

Information that is made available via this modal is as follows:

  • Online Status
  • Organization ID
  • Installer ID
  • Sensor ID
  • Enrolment Date
  • Last External IP
  • Last Internal IP
  • Last Time Connected
  • Hostname
  • Platform
  • Network Isolation Status
  • Tags 

The laptop icon launches the user into Live View. Live view opens up a secure tunnel to the endpoint - through the LimaCharlie Cloud - with which the user can interact in real-time. It is a complicated set of capabilities launched in its own view and is covered in detail later in this course.

Using the Live View operators are able to:

  • Get general information about the endpoint.
  • Control the verbosity of telemetry being sent by the specific endpoint.
  • Use a web console to query the sensor in real-time.
  • Start a live feed of events taking place on the endpoint.
  • Examine the processes and perform basic admin on them.
  • Interact with the endpoint’s file system 

The clock icon launches the user into LimaCharlie's Historical Insight where users can visually navigate and search across all data for the given endpoint. This feature is not available without retention enabled. Historical Insight is a complicated set of capabilities launched in its own view and is covered in detail later in this course.


Installation Keys

An installation key is used when installing a new sensor. It allows you to automatically tag sensors coming from a specific key. If the key is leaked, deleting it will prevent any new sensors attempting to enroll using it from associating with your organization.

Installation keys are displayed as line items with each column described below.

Description: the description for the key entered at the time of creation.

Tags: displays tags that will automatically be assigned to the sensor when enrolling with the given Installation Key.

Sensor Key: the copy icon in this column will copy the Sensor Key to the operators clipboard when clicked.

Chrome Key: the copy icon in this column will copy the Chrome Key to the operators clipboard when clicked. This key is specific to the Chrome browser and Chrome OS.

To add a new Installation Key click the plus icon in the upper right corner and fill in the modal form. Add a description and any tags that you want associated with this key. Tags set while creating the Installation Key cannot be removed.

 


Outputs

The LimaCharlie endpoint agent produces telemetry in well documented JSON that can be output anywhere. LimaCharlie has modules supporting Slack, Google Cloud Storage, S3, SFTP, Syslog, SMTP and SCP. 

Any  number of output modules can be active at one time and the verbosity for each can be adjusted (event, detect, audit and deployment levels). This means you can stream to two different syslog destinations using the Syslog Output module and then send the same data to cold storage over an Scp Output module.

To add a new Output simply click the plus icon in the upper right corner and complete the form for the specific module you wish to use. The data required varies depending on the type of module and in depth look at each is beyond the scope of this guide. For further information on setting up a particular module please refer to the documentation.

Documentation on setting up the various output connections can be found here.

In the user interface Outputs are displayed as line items in a list with each column defined as follows.

Name: the name of the output given at creation time.

Module: the class of module used to broker the output connection.

Stream: the selected verbosity level for telemetry being sent to this Output.

DR Rules

D&R Rules shows the list of Detection & Response rules that are applied to the organization. The D&R Rules are displayed as line items and details about the D&R Rule along with its operational status.

With LimaCharlie users can create custom Detection & Response rules. The Detection component is a scriptable rule system that can be chained with multiple conditions that will match certain events. When the Detection component matches the Response component is actioned. Responses are actioned in real-time by the sensor via a secure semi-persistent TLS connection. The Detection & Response engine automates your ability to investigate, mitigate or apply tags and more using serverless functions. Round-trip time from detection until the response action is typically under 100ms.

Documentation on custom D&R rules can be found here.

D&R Rules are displayed as line items with each of the columns defined below.

Name: the given name of the D&R Rule

Details: hovering over this field triggers a tool-tip with the body of the D&R Rule being displayed.

Edit: the edit icon opens the D&R Rule GUI editor.

Copy: copies the D&R Rule to the users clipboard.

Delete: destroys the rule.

To create a new D&R Rule click on the plus icon in the upper right corner. A rule building modal will launch and the user can choose from various ways to create a D&R Rule. There are three modes for the rule builder outlined as follows.


Easy: the easy assist mode can be used to create simple rules with a couple of clicks. The source code for the rule is generated and displayed in the Advanced tab where the user can further alter it.

Assist: is a slightly more advanced rule builder that allows users to create complex detection logic and to chain boolean logic together 

Advanced: the advanced tab of the rule builder allows users to write their own D&R Rules in raw YAML.


D&R Rules are a big and comp[lex topic which will be covered in depth in a later course. Documentation on D&R RUles can be found here.

Exfil Control

The Exfil Control view allows the user to control what event types are sent to the cloud and under what conditions. Control of the telemetry filtering is done through the use of rules.

By default, LimaCharlie sensors send events to the cloud based on a standard profile that includes events like NEW_PROCESS, DNS_REQUEST etc.

If you enable the Exfil Service, this default profile is replaced by a custom set of rules you define.

The rule customization is done in two parts:

  1. The Event Rules, which simply define a list of events that should be sent to the cloud based on platform and tags.

  2. The Watch Rules, which define additional events that should be sent to the cloud based on the content of each individual event.

The Watch Rules allow you to specify a platform and tag to select which sensors the rule applies to, plus these elements:

  • Event: the specific event type that should be evaluated, like MODULE_LOAD.

  • Path: the path within the event component whose value should be evaluated, like FILE_PATH.

  • Operator: the type of evaluation/comparison that should be done between the value at Path in the event and the Value.

  • Value: the value used in the comparison.

Performance Mode

Although not strictly related to Exfil, the sensor's Performance Mode can also be set as a rule on top of interactively.

This is done by modifying Performance Rules through the web app or REST interface. Any sensor matching a rule will have its performance mode enabled.

Documentation for Exfil Control can be found here: https://doc.limacharlie.io/en/master/exfil/

Replay

Replay enables historical threat hunting meaning that users can run D&R Rules against historical telemetry or log files.

This allows an analyst to select a date range and search an endpoint’s history, or scan across the history of the entire fleet, for indicators of compromise (IOC). Almost any detection that can be written can be applied retroactively against historical telemetry.

The user interface requires that a user select a date range for the scan, D&R rule to use and finally an endpoint or set of endpoints to scan. Once this data has been provided the user initiates the process and the results are displayed in an incident card.

Documentation for Replay can be found here: https://doc.limacharlie.io/en/master/replay/


File/Reg Integrity Monitoring

File/Reg Integrity Monitoring
This view allows the user to control all aspects of file and registry integrity monitoring through the use of rules. Rules define which file path patterns and registry patterns should be monitored for changes for specific sets of hosts.

Filter tags are tags that must ALL be present on a sensor for it to match (AND condition), while the platform of the sensor much match one of the platforms in the filter (OR condition).

Patterns are file or registry patterns, supporting wildcards (*, ?, +). Windows directory separators (backslash, "\") must be escaped like "\".

Linux

Integrity monitoring is partially supported on Linux. Specified file path expressions are actively monitored via inotify (as opposed to MacOS and Windows where kernel passively monitors).

Due to inotify limitations, paths with wildcard are less efficient and only support monitoring up to 20 subdirectories covered by the wildcard. In addition to this, the path expressions should specify a final wildcard of * when all files under a directory need to be monitored. Omitting this * will result in only the directory itself being monitored.

Documentation on File/Reg Integrity Monitoring can be found here: https://doc.limacharlie.io/en/master/integrity/

Yara

The Yara service is designed to help you with all aspects of Yara scanning. It takes what is normally a manual piecewise process, provides a framework and automates it.

Once configured Yara scans can be run on demand for a particular endpoint or continuously in the background across your entire fleet.

Note that Yara configurations are synchronized with sensors every few minutes.

There are three main sections to the YARA job.

Sources

This is where you define the source for your particular YARA rule(s). Source URLs can be either a direct link to a given YARA rule or ARLs].

An example of setting up a source using this repo: Yara-Rules

For Email and General Phishing Exploit rules we would link the following URL, which is basically just a folder full of .yar files.

https://github.com/Yara-Rules/rules/tree/master/email

Giving the source a name and clicking the Add Source button will create the new source.

Rules

Rules define which sets of sensors should be scanned with which sets of YARA signatures (or sources).

Filter tags are tags that must ALL be present on a sensor for it to match (AND condition), while the platform of the sensor much match one of the platforms in the filter (OR condition).

Scan

To apply YARA Sources and scan an endpoint you must select the hostname and then add the YARA Sources you would like to run as a comma separated list.

Response

The Responder service is able to perform various incident response tasks for you by performing a sweep.

A sweep is an in-depth look through the state of a host and will highlight activity that is suspicious. This provides you with a good starting position when beginning an investigation. It allows you to focus on the important things right away.

The types of information returned by the sweep is constantly evolving but you can expect it to return the following information:

  • A full list of processes and modules

  • A list of unsigned binary code running in processes

  • Network connections with a list of processes listening and active on the network

  • Hidden modules

  • A list of recently modified files

  • Unique or rare indicators of compromise

Search

Through the web application users can search all telemetry, logs, portable executables and pre-fetch files that are ingested by LimaCharlie. The telemetry and artifacts collected are all  indexed across common indicators of compromise. Along with this, all events are marked with a special set of IDs, referred to as atoms, that can be used to traverse process trees.

This feature requires retention (Historical Insight) to be enabled.

Searching is powered by BigQuery and this enables users to search up to a year’s worth of telemetry across entire fleets with results available in moments.

You can query for the following kind of information:

Prevalence Summary: this will give you three metrics. The number of hosts where the indicator has been seen today, this week and this month. This is great to give you information at a glance about an indicator. For example, an executable seen twice today and never this week and this month is more suspicious than one seen over hundreds of hosts for the whole month.

Locations: this will show you where this indicator has been seen as well as the first and last time it was observed on each host. This is useful during the incident response process to scope where specific suspicious behavior has been observed.

Artifact Collection

The Artifact Collection system allows you to ingest artifact types like:

  • Plain text logs (syslog for example)

  • Windows Event Logs

  • PCAPs

  • Windows Prefetch files

  • Windows PE (executables) files

  • Zeek (previously Bro)

  • Full memory dumps

  • Generic JSON

  • OLE (MS Word, Excel etc)

Artifacts can be ingested from hosts running a LimaCharlie sensor or they can be pushed to the LimaCharlie platform via a REST interface.

The artifacts are retained and made available with a custom retention period. Ingested artifacts are also indexed similarly to LimaCharlie events. This means that you can search all of your artifacts for the last year for Indicators like IP Addresses, Domain Names, User Names, Hashes etc.

This in turn makes it possible for you to be looking at sensor data, identity an IP of interest, and launch a quick search to see if this IP has been observed in any artifacts over the past year. If it has, with one click you can visualize the relevant artifact entries to assist you in your investigation.

We call this "artifact operationalization". It is not mean to be a general viewing and querying tool like Splunk, but as a tactical tool providing you with critical answers as you need them during security operations.

Note that Artifact Collection configurations are synchronized with sensors every few minutes.

Platform Logs

LimaCharlie produces several log types that are made accessible through the web application.

Management Logs: the management logs are generated by the LimaCharlie Cloud and report activity at an account and organization level. Who requested quota, gained access to the organization, billing etc.

Error Logs: system and configuration errors.

Audit Logs: read only logs of system activity that can be used to trace operator behaviour.

Sensor Downloads

From this interface users can download sensor builds for various architectures and manage the version of the sensor currently running on their endpoints.

All versions of the sensor are backwards compatible and the user can roll back versions with the click of a button in the unlikely scenario that a version has an unexpected negative impact on the endpoint.

The agent is supported on Windows XP and up, MacOS, Linux, Docker, Chrome and Chrome OS, with distributions across x86, ARM and MIPS architectures (32 & 64 bit). Custom builds of the agent for OpenBSD and Solaris can be made upon request.

There is also an agent for Chrome and Chrome OS which encompasses a subset of the functionality. This agent can be run on Chromebooks or as a stand-alone in the browser. When used in parallel with the primary agent visibility into URL requests is obtained. The Chrome agent is included in the base fee for the primary agent.

You can download the Chrome agent from the Chrome store.

Add Ons (Extensions)

The Add-ons marketplace is a place where users can subscribe to various capabilities as provided by LimaCharlie or other users.

Several categories of capabilities exists like Detections and Lookups:

  • Detections are components you can use as part of D&R rules.

  • Lookups are lists of values to match in incoming LC data along with metadata for each value. They are use as part of D&R rules as well.

Users can create their own add-ons privately which gives them access to enable those rules in organizations they are members of, or publicly where anyone in the marketplace can subscribe.

Using

In order to use add-ons, you must first subscribe to them with an organization. To do this, go to the "Add-ons" menu in your organization, then go to the category and find the add-on you want to subscribe to. Click on the subscribe switch.

After a second you will get a confirmation the add-on has been enabled for your organization.

Now go to the "D&R Rules" menu and create a rule using the add-on by using the relevant operation (external or lookup) and specifying the resource as the add-on URL.

The add-on URL is simply a path like an HTTP URL, but starting with lcr:// (for LimaCharlie Resource) The next component of the path is the category of add-on (lookup or detection) and finally the add-on name. So a path can look something like this:

lcr://detection/win-suspicious-exec-location

 

Creating

Users can create their own add-ons and optionally share them in the marketplace.

To create a new add-on, do it from the dashboard of LimaCharlie.io. When doing so, especially when marking the add-on as public, keep these in mind:

  • Fill in as many of the fields as possible.

  • Make the purpose and usage of the add-on clear for users not aware of the capability.

  • If the add-on requires a lot of background information, put it in the website link rather than the description.

  • Your email address will be included in the add-on description, if you plan on publishing many rules you may want to create a limacharlie.io account specifically for this purpose.

  • The platforms flags are not enforced but are a courtesy to the users to let them know where they can expect your add-on to work.

  • Subscribers to your add-on will NOT have access to the exact implementation, this helps you protect specific Intellectual Property you may have, but it also means you need to be clear about the exact capabilities in your description and website.

  • Add-ons remain your property, but may be evaluated and approved / dismissed due to quality or performance concerns. If you are not sure, contact us at support@limacharlie.io.

  • If implementing a detection, include "filtering" of the relevant event types to what is strictly required for the detection, but avoid filtering on organization-defined labels as they change from organization to organization. If you do use labels, describe the exact usage in the description.

  • Before making an add-on public, make sure to test it.

Keep in mind that creating an add-on does not immediately grant organizations you're a member of access to it. After creating it, you must go in the organization you want to access it from, go to the Add-ons section and "subscribe" to your add-on. This will grant the organization access and then you can begin using it as part of D&R rules.

Lookups

Creating a lookup add-on enables you to create a list that you can use as part of D&R rules. Once in place, you can refer to it using the op: lookup D&R rule with a reference to your add-on looking like resource: lcr://lookup/my-lookup-name.

Lookups support a few structures.

  • Newline-separated values.

  • JSON dictionary where keys are the elements of the lookup and the values are the metadata associated.

  • YAML dictionary where keys are the elements of the lookup and the values are the metadata associated.

  • OTX JSON Pulse.

  • MISP JSON Feed.

Here is an example of this complex format:

evil.com: some evil website, definitely bad

example.com:

  source: my-threat-intel

  risk: high

  contact: email threatintel@mycorp.com immediately if spotted

 

When uploaded, the data for the lookup can be provided in three different ways:

  1. As data literal in the upload API.

  2. As a URL callback, where your data is a URL like https://www.my.data.

  3. As an Authenticated Resource Locator (ARL) (the preferred method)

The maximum size of a lookup is 15MB through the REST API and 512KB through the web interface.
