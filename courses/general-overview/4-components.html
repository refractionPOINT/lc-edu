LimaCharlie delivers infrastructure using a services model. The cornerstone of the LimaCharlie platform is the Detection & Response component on which the majority of services are built.

There are two main modes of operation. The simplest is running the LimaCharlie Agent as middleware and piping the data into an existing pipeline. When running as middleware LimaCharlie never writes any telemetry to disk.

The second class of functionality is a superset of the first class and becomes available when Storage is enabled. Storage is optional telemetry storage with a one year rolling window of history. In this mode all telemetry is captured and indexed across common indicators of compromise.

Running as middleware LimaCharlie enables:
  • Detection & Response
  • Live View
  • Payloads & File Retrieval
  • Data Forwarding

The superset of functionality - with Storage enabled - provided to LimaCharlie users:

  • Telemetry Storage
  • Artifact Collection
  • Search & Historical Threat Hunting
  • Integrations
  • Community Resources
  • Public & Private Extensions
  • Product Development

Details on each of these capabilities and functionality can be found below.

Detection & Response

The LimaCharlie Agent monitors over 70 different event types as they take place on the endpoint. JSON Telemetry describing these events is sent over a semi-persistent, real-time connection to the LimaCharlie Cloud where it is processed.

Detection & Response (D&R) describes the process by which telemetry generated by a given endpoint agent triggers a detection inside of the LimaCharlie Cloud and a response action takes place. Round-trip time from when an event takes place that triggers a detection to a response being actioned on the endpoint is generally around 100ms.

D&R Rules are simple expressions that describe conditions that will trigger a detection when matched. Once a D&R Rule is written it can be pushed to the LimaCharlie Cloud where it will instantly be applied to every endpoint in the fleet. It can help with conceptualization to think of D&R Rules as akin to AWS Lambda or Google Cloud Functions.

Live View

Live View allows the user to interact with the endpoint in real-time. Users interact with the endpoints as if they were an extension of the cloud and with Live View users can:

  • Adjust the telemetry for a given endpoint.
  • Send console commands.
  • Stream a feed of events taking place on the endpoint in real-time.
  • List the running processes.
  • View modules.
  • Get memory strings.
  • Get memory maps.
  • Pull system information, kill suspend and resume processes.

Payloads & File Retrieval

The agent maintains a semi-persistent, real-time connection to the LimaCharlie Cloud which allows the user to retrieve files of any size from the endpoint. Data is streamed in approximately 10MB chunks and sent to a bucket in the LimaCharlie Cloud for reassembly.

Conversely, and under very specific permissions, users can send an executable down to the host for execution.These payloads can be any executable. The main use case is to run something with specific functionality not available directly from LimaCharlie. For example: custom executables provided by another vendor to cleanup a machine, forensic utilities or firmware-related utilities.

The payloads feature is powerful and should be treated as such. Users are encouraged to look at LimaCharlie’s native functionality first as it has several advantages:

  • Better performance.
  • Data returned is well structured JSON.
  • Can be tasked automatically
  • Detection & Response rules can be created from their data.
  • Data returned is indexed and searchable.

Data Forwarding

All telemetry produced by the LimaCharlie agent can be forwarded to any location. Most common technologies are supported such as Syslog, SFTP, AWS S3, Google Storage, SMTP and more. Users can set up complex forwarding rules for different types of telemetry and alerts.

Forwarding is done in real-time or near real-time depending on forwarding methodology.
If storage is not enabled then the telelemtry being forwarded is done so on the wire and never written to disk as it transits through the LimaCharlie infrastructure.

Artifact Collection

LimaCharlie can ingest any files from the endpoint as well as files pushed through the API/CLI/SDK.

Log files can be uploaded on an ongoing basis and have detection logic applied to them as they are being processed. Detection logic can also be applied to historical logs across the entire fleet.LimaCharlie can consume logs from any OS. Logs can be unstructured (no parsers necessary), pcap or Windows logs. Another typical use case sees users downloadportable executables and pre-fetch files which can also have detection logic applied to them.

Information on log ingestion can be found in the documentationhere, and information on log specific DR rules can be found here.

Search & Historical Threat Hunting

All telemetry, logs, portable executables and pre-fetch files that are ingested by LimaCharlie are indexed across common indicators of compromise. Along with this, all events are marked with a special set of IDs, referred to as atoms, that can be used to traverse process trees.

Searching is powered by BigQuery and this enables users to search up to a year’s worth of telemetry across entire fleets with results available in moments.

LimaCharlie also uses the concept of targets with D&R rules which means that users can create detection logic that can be applied to log files as they are ingested or over historical log data.

Integrations

LimaCharlie exposes a full-featured and well-documented REST API along with providing a Python CLI and SDK.

LimaCharlie enables automation and integration with other systems and feeds. An HTTPS-based data stream - referred to as PULL mode - can be used to create real-time integrations.

Along with an Integration framework, LimaCharlie offers two pre-built integrations.

VirusTotal

Use LimaCharlie with Virus Total. Bring your own key and have hashes from your endpoints checked on VT automatically. Make it part of your Detection & Response rules to automate response in real-time. Pool your queries and save money.

Humio

Use LimaCharlie with Humio to store telemetry data. Humio is a solution built specifically for aggregating, exploring, reporting and analyzing data in real-time. It gathers log data from a range of sources and can be deployed in both cloud and on-premise environments.

Backstory

LimaCharlie has partnered with Chronicle Security to deliver an integration with their Backstory security analytics platform.

The LimaCharlie agent contributes valuable endpoint telemetry and capability to Backstory’s enterprise security offering. The integration with Backstory will allow for the global correlation between log and endpoint telemetry providing a unique global perspective and access to threat intelligence not found anywhere else.

Community Resources

Leveraging the integration and extension friendly nature of the platform, LimaCharlie has built-in support for several Community Resources.

YARA

LimaCharlie can now use YARA rules to scan external files as they are ingested. This new feature comes on the heels of our historical threat hunting service (Replay) which allows users to search up to a year’s worth of telemetry data across a fleet using Detection & Response rules. We believe that these two features combined make for a powerful set of detection tools. The possibilities for new forensic and defensive capabilities are near limitless.

Details on how this new YARA feature fits within the content of the existing Detection & Response framework can be found here:https://doc.limacharlie.io/en/master/dr/#yara

Sigma

The Sigma rule set is an open source signature format that can describe relevant log events in a straightforward manner. The rule format itself is very flexible and applicable to any type of log file. The main purpose of Sigma is to provide a structured form with which researchers and analysts can describe and share detections methods with the wider community.

Zeek

LimaCharlie users can now register for the Zeek Service. With this service enabled any PCAP files uploaded to LimaCharlie (either automatically through an agent, the API or through the CLI) is automatically processed and “exploded” using Zeek.

The Zeek log format gets normalized to JSON, which in turn allows D&R rules to be applied to those logs.

https://www.limacharlie.io/blog/2020/2/6/introducing-zeek-network-monitoring

MISP & OTX Pulse Formats

LimaCharlie supports the MISP JSON format. If you create a Lookup pointing to MISP JSON, the format will be parsed and the lookup will associate the MISP IDs as the Attribute metadata. LimaCharlie similarly supports the JSON OTX Pulse format from AlienVault.

Public & Private Extensions

Authenticated Sources

Many of the services we provide facilitate outside resources. Today we are happy to announce support for the consumption of external resources that require authentication. YARA rules, threat feeds and other external resources can now be fetched from private Github repos and other protected sources.

Live Sweep

A Live Sweep is an in-depth look through the state of a host. The sweep will highlight parts of the activity that are suspicious. This provides you with a good starting position when beginning an investigation. It allows you to focus on the important things right away.

The types of information returned by the sweep is constantly evolving but you can expect it to return the following information:

  • A full list of processes and modules
  • A list of unsigned binary code running in processes
  • Network connections with a list of processes listening and active on the network
  • Hidden modules
  • A list of recently modified files
  • Unique or rare indicators of compromise
