Endpoint Detection & Response (EDR)

The first core technology available in LimaCharlie was the EDR.

The EDR provides a whole suite of powerful capabilities and is built off a lightweight, cross-platform agent.

The LimaCharlie endpoint agent has the best cross-platform support in the industry and can be used to bring the most diverse fleets into a single detection stream. The agent runs on 32-bit Windows all way back to Windows XP SP II through to the most modern version of 64-bit Windows. It can run on all flavors of Linux both 32-bit and 64-bit. Builds for MacOS (Intel & M1) are also available. Builds for Solaris and BSD that can be produced on request. The lightweight footprint of LimaCharlie’s agent makes it well suited for IoT and builds across x86, ARM and MIPS architectures are available for all platforms.

The agent is written in C and then compiled for each platform and architecture it runs on, which means that it has true feature parity across all operating systems. The only exceptions are platform specific functions, such as monitoring Windows registry operations, etc.

When the agent is installed and running it is referred to as a sensor. On disk the agent is approximately 500kb in size but that varies depending on which platform it is compiled for. While running it consumes less than 1% CPU but does spike very briefly when certain events, such as an application starting up, take place. LimaCharlie is able to pack so much power into such a small program because it treats the agent as an extension of the cloud by utilizing a true real-time persistent TLS connection. The round trip time from an event being detected to the time a response is actioned on the endpoint is generally less than 100 milliseconds.

LimaCharlie also offers the industry's only Chrome OS agent. The agent can be used as a standalone on machines running Chrome OS or as a sidecar agent in the browser where it can do HTTP inspection removing the need for SSL interception.

The Detection & Response Engine behind the EDR capability plays a central role in coordinating between the different core technologies and lays the foundation for automating operations at scale.

Telemetry is produced for evaluation by the Detection & Response Engine via the sensor that monitors Events taking place on the endpoint. The LimaCharlie sensor monitors over 75 Events taking place on the endpoint. A full list of Events can be found here.

Complete endpoint telemetry is stored in a one year rolling window. All telemetry is indexed and searchable across common indicators of compromise.

Detection rules are written against telemetry and when a positive match occurs a Response action is initiated.

Response actions can take many forms and are often used for automating between different parts of the LimaCharlie platform. Documentation on Detection & Response (DR) rules can be found here. A list of example response actions is as follows.

  • Isolating the host from the network.
  • Killing the process tree that triggered the alert.
  • Notifying the response team via a pager duty mechanism.
  • Initiate full PCAP capture from the network.
  • Tag the offending machine as suspicious.
  • Run forensics and trigger a memory dump.
  • And much much more.

Artifact Collection

Artifact Ingestion allows externally generated artifacts to be ingested and processed by the LimaCharlie Detection & Response Engine.

The list of artifacts that are supported is constantly growing and includes the following. If there is interest in processing a different kind of artifact please get in touch. Adding new file formats is generally an easy process and can usually be turned around in a matter of days.

  • Windows Event Logs
  • Plain text logs (syslog for example)
  • PCAPs
  • Windows Prefetch files
  • Windows PE (executables) files
  • Zeek (previously Bro)
  • Full memory dumps
  • Generic JSON
  • OLE (MS Word, Excel etc)
  • Windows MFT CSV Listing
  • Apple Binary/XML plists

Artifacts can be ingested by LimaCharlie in several ways. Full documentation on Artifact Ingestion can be found here.

  • Directly from the agent
  • Through the REST API
  • Through the Python CLI or SDK

Upon ingestion, artifacts are converted into a common JSON format that is common to all telemetry in the LimaCharlie platform. This JSON, along with the original artifact, are stored for a custom retention period defined by the user. This means all the user's artifacts can be searched over the custom retention period for indicators of compromise such as:

  • Domain
  • IP Address
  • File Hash
  • File / Process Path
  • File / Process Name

This also means that the same Detection & Response Engine used to power LimaCharlie’s EDR can be applied to artifacts. These DR rules can be applied to Artifact Ingestion in real-time or over historical data using a service called Replay.

We do not consider Comms to be a SIEM but do think it can replace them for the majority of use cases. Features Worth Talking About

Within the product pillars there are many integrations and capabilities that are worth calling out on their own. An extensive but not exhaustive list is as follows.

Live View

Live View allows the user to interact with the endpoint in real-time. Users interact with the endpoints as if they were an extension of the cloud and with Live View users can:

  • Adjust the telemetry for a given endpoint.
  • Send console commands.
  • Stream a feed of events taking place on the endpoint in real-time.
  • List the running processes.
  • View modules.
  • Get memory strings.
  • Get memory maps.
  • Pull system information, kill suspend and resume processes.

Payloads & File Retrieval

The agent maintains a semi-persistent, real-time connection to the LimaCharlie Cloud which allows the user to retrieve files of any size from the endpoint. Data is streamed in approximately 10MB chunks and sent to a bucket in the LimaCharlie Cloud for reassembly.

Conversely, and under very specific permissions, users can send an executable down to the host for execution. These payloads can be any executable. The main use case is to run something with specific functionality not available directly from LimaCharlie. For example: custom executables provided by another vendor to cleanup a machine, forensic or firmware-related utilities.

The payloads feature is powerful and should be treated as such. Users are encouraged to look at LimaCharlie's native functionality first as it has several advantages:

  • Better performance.
  • Data returned is well structured JSON.
  • Can be tasked automatically
  • Detection & Response rules can be created from their data.
  • Data returned is indexed and searchable.

Data Forwarding

All telemetry produced by the LimaCharlie agent can be forwarded to any location. Most common technologies are supported such as Syslog, SFTP, AWS S3, Google Storage, SMTP and more. Users can set up complex forwarding rules for different types of telemetry and alerts.

Forwarding is done in real-time or near real-time depending on forwarding methodology.
If storage is not enabled then the telemetry being forwarded is done so on the wire and never written to disk as it transits through the LimaCharlie infrastructure.

Artifact Collection

LimaCharlie can ingest any files from the endpoint as well as files pushed through the API/CLI/SDK.

Log files can be uploaded on an ongoing basis and have detection logic applied to them as they are being processed. Detection logic can also be applied to historical logs across the entire fleet. LimaCharlie can consume logs from any OS. Logs can be unstructured (no parsers necessary), pcap or Windows. Another typical use case sees users download portable executables and pre-fetch files which can also have detection logic applied to them.

Information on log ingestion can be found in the documentation here, and information on log specific DR rules can be found here.

Search & Historical Threat Hunting

All telemetry, logs, portable executables and pre-fetch files that are ingested by LimaCharlie are indexed across common indicators of compromise. Along with this, all events are marked with a special set of IDs, referred to as atoms, that can be used to traverse process trees.

Searching is powered by BigQuery and this enables users to search up to a year’s worth of telemetry across entire fleets with results available in moments.

LimaCharlie also uses the concept of targets with D&R rules which means that users can create detection logic that can be applied to log files as they are ingested or over historical log data.

Integrations

LimaCharlie exposes a full-featured and well-documented REST API along with providing a Python CLI and SDK.

LimaCharlie enables automation and integration with other systems and feeds. An HTTPS-based data stream - referred to as PULL mode - can be used to create real-time integrations.

Along with an Integration framework, LimaCharlie offers two pre-built integrations.

VirusTotal

Use LimaCharlie with Virus Total. Bring your own key and have hashes from your endpoints checked on VT automatically. Make it part of your Detection & Response rules to automate response in real-time. Pool your queries and save money.

Humio

Use LimaCharlie with Humio to store telemetry data. Humio is a solution built specifically for aggregating, exploring, reporting and analyzing data in real-time. It gathers log data from a range of sources and can be deployed in both cloud and on-premise environments.

Backstory

LimaCharlie has partnered with Chronicle Security to deliver an integration with their Backstory security analytics platform.

The LimaCharlie agent contributes valuable endpoint telemetry and capability to Backstory’s enterprise security offering. The integration with Backstory will allow for the global correlation between log and endpoint telemetry providing a unique global perspective and access to threat intelligence not found anywhere else.

Community Resources

Leveraging the integration and extension friendly nature of the platform, LimaCharlie has built-in support for several Community Resources.

YARA

LimaCharlie can now use YARA rules to scan external files as they are ingested. This new feature comes on the heels of our historical threat hunting service (Replay) which allows users to search up to a year’s worth of telemetry data across a fleet using Detection & Response rules. We believe that these two features combined make for a powerful set of detection tools. The possibilities for new forensic and defensive capabilities are near limitless.

Details on how this new YARA feature fits within the content of the existing Detection & Response framework can be found here.

Sigma

The Sigma rule set is an open source signature format that can describe relevant log events in a straightforward manner. The rule format itself is very flexible and applicable to any type of log file. The main purpose of Sigma is to provide a structured form with which researchers and analysts can describe and share detections methods with the wider community.

Zeek

LimaCharlie users can now register for the Zeek Service. With this service enabled any PCAP files uploaded to LimaCharlie (either automatically through an agent, the API or through the CLI) is automatically processed and “exploded” using Zeek.

The Zeek log format gets normalized to JSON, which in turn allows D&R rules to be applied to those logs.

MISP & OTX Pulse Formats

LimaCharlie supports the MISP JSON format. If a Lookup pointing to MISP JSON is created, the format will be parsed and the lookup will associate the MISP IDs as the Attribute metadata. LimaCharlie similarly supports the JSON OTX Pulse format from AlienVault.

Public & Private Extensions

Authenticated Sources

Many of the services we provide facilitate outside resources. Today we are happy to announce support for the consumption of external resources that require authentication. YARA rules, threat feeds and other external resources can now be fetched from private Github repos and other protected sources.

Live Sweep

A Live Sweep is an in-depth look through the state of a host. The sweep will highlight parts of the activity that are suspicious. This provides a good starting position when beginning an investigation. It allows for focus on the important things right away.

The types of information returned by the sweep is constantly evolving but one can expect it to return the following information:

  • A full list of processes and modules
  • A list of unsigned binary code running in processes
  • Network connections with a list of processes listening and active on the network
  • Hidden modules
  • A list of recently modified files
  • Unique or rare indicators of compromise
