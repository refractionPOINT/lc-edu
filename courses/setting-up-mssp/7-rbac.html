To recap, we have defined two classes of objects when thinking about LimaCharlie deployments at a high level.

Users

A user is basically a unit of access and will usually represent a person and their credentials but it could just be a set of credentials used for a special types of access.

Organizations

This is the main organizational unit for companies or groups. An MSSP will generally have multiple organizations under their control. All data and access between organizations is siloed. Billing is also done per organization.

Users can be grouped together with a common permission scheme called Organization Groups.

Organization Groups

Organization Groups allow you to grant permissions to a set of users on a group of organizations.

Permissions granted through the group are applied on top of permissions granted at the organization level. Their permissions are additive, and a group cannot be used to subtract permissions granted at the organization level.

Organization Group Owners are allowed to manage the Organization Group, but are not affected by the permissions. Members are affected by the permissions but do not have the ability to modify the Organization Group.

API Keys

There are two types of API keys available with LimaCharlie.

Org Level API Key

  • An org level API key only applies to the specific organization it was created for and the permissions can be set per-key. That is, a virtually unlimited number of keys with vastly different permissions can be created per organization.

User Level API Key

  • A user level API key can be applied to all organizations and permissions that the given user has. That is to say, user API keys that are created automatically inherit the permissions of the user that is creating them. These keys are powerful and should be used cautiously.

Because LimaCharlie is multi-tenant, it is advantageous to create a special CI/CD user account to use for orchestration. For our purposes we will refer to this account as the root account.
